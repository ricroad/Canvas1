# 常见问题 Q&A

## 1. 这个项目本质上是什么

答：它是一个节点式 AI 创作平台，而不是单次提交提示词的页面工具。关键证据是画布、连线、结果节点、派生链路和项目级持久化都已经是核心能力。

## 2. 为什么要做成节点画布

答：因为目标任务天然是多步骤试错。用户要保留中间结果、从结果继续派生、组合不同链路，这些都比线性表单更适合画布。

## 3. 现在有多少种正式节点

答：当前是 13 种，定义在 `canvasNodes.ts`，注册行为在 `nodeRegistry.ts`。

## 4. 为什么结果节点单独建模

答：因为当前系统已经支持 batch 输出和从具体结果继续派生。生成节点和结果节点分开后，语义更稳定，也更适合恢复和回退。

## 5. `imageNode` 和 `imageResultNode` 的区别是什么

答：前者负责参数与生成动作，后者负责结果集与选中结果。两者不是重复，而是“动作”和“产物”的分层。

## 6. `videoGenNode` 和 `videoResultNode` 也是同样思路吗

答：是。视频链路也已经采用生成节点发起 batch、结果节点承载 variants 的模式。

## 7. `nodeRegistry.ts` 为什么这么重要

答：因为它决定了节点默认数据、菜单可见性和连线能力，是节点系统的单一真相源。

## 8. 新增一个节点为什么不是只写个组件

答：因为节点要被系统正式认识。通常还要补类型、注册表、渲染注册，以及可能的数据流和持久化规则。

## 9. 哪些节点是用户不能直接从菜单创建的

答：典型有 `imageResultNode`、`videoResultNode`、`storyboardNode`、`exportImageNode`、`groupNode`。这类节点更像系统流程产物。

## 10. live resolve 是什么

答：下游节点在运行时动态沿边取上游数据，而不是连线时把字段抄一份下来。

## 11. connection-time copy 是什么

答：在连线发生时直接把上游某个值写进下游节点数据。当前 `sceneComposerNode.inputImageUrl` 就是这种典型用法。

## 12. 为什么两种机制要同时存在

答：因为不同节点的语义不同。普通生成节点更适合动态解析，特殊嵌入式节点有时更适合拿连接瞬间的快照。

## 13. `canvasStore` 和 `projectStore` 的边界是什么

答：`canvasStore` 管图本身和运行时操作，`projectStore` 管项目生命周期与写盘协调。一个偏编辑态，一个偏项目态。

## 14. `settingsStore` 只是主题设置吗

答：不是。它还包含 API key、Kling 设置、视频并发、价格显示、下载路径、边路由模式等运行配置。

## 15. 这个项目是自动保存吗

答：是，而且是分通道自动保存。整项目快照和视口快照分开处理。

## 16. 为什么视口要单独保存

答：因为平移缩放很高频，如果每次都重写整项目，会影响交互体验。

## 17. 图片和视频为什么要做引用池

答：为了避免在节点与历史里重复展开大体量媒体引用，降低序列化体积。

## 18. Web 版现在能做什么

答：能做项目存储、部分文本能力和部分浏览器可承载的图片操作，但不是桌面端完整镜像。

## 19. Web 版现在不能等价做什么

答：至少图片生成和视频生成当前都不是等价能力，代码里已明确提示 desktop-only。

## 20. Tauri 在这里的意义是什么

答：它不是壳子而已，而是桌面能力边界。很多项目核心能力实际通过 Rust command 执行。

## 21. Rust 层大致负责什么

答：项目持久化、图片处理、模型调用、系统信息、更新检查等原生能力。

## 22. 新增模型通常要改哪些地方

答：至少是前端模型定义，加上必要的后端适配。只改前端下拉通常不够。

## 23. 新增 provider 为什么更贵

答：因为它通常横跨前端 provider、模型定义、设置链路和 Rust provider 实现。

## 24. Scene Composer 为什么总被单独讨论

答：因为它是特殊嵌入式节点，不像普通节点那样只靠标准输入框和 resolver 运作。

## 25. 什么需求最容易被低估

答：任何涉及“结果派生”“自动同步”“项目恢复”“桌面与 Web 一致性”的需求都容易被低估。

> **📌 PM 视角：** 这类问答最核心的用途不是背答案，而是帮你在提需求前先把问题问对。只要你能先区分节点、结果、持久化和桥接，和工程师对齐会快很多。
