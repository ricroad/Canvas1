# Infinite Canvas 一页纸总览

## 这是什么产品

一句话版本：**Infinite Canvas 是一个基于节点画布的本地 AI 创作工作台**，把图片上传、AI 图像生成/编辑、剧本拆解、分镜规划、视频生成和结果管理串成一条可视化工作流。

三句话版本：

1. 用户不是在单个表单里反复提交，而是在画布上摆节点、连线、派生结果、保留中间产物。
2. 系统里不只有“生成节点”，还有独立的结果节点，例如 `imageResultNode` 和 `videoResultNode`，用于承载批次输出和后续派生。
3. 它不是单纯前端页面，而是一个分层平台：React UI + Zustand 状态 + application 服务 + command 桥接 + Tauri/Rust + SQLite 或 IndexedDB。

## 当前系统的核心能力

- 节点画布：基于 `@xyflow/react` 的无限画布，支持拖拽、缩放、连线、结果派生、分组、历史回退。
- 图片链路：上传图片、图生图、文生图、多参考图、裁剪、标注、分镜切图。
- 分镜链路：剧本上传、LLM 生成分镜提示词、分镜批量出图、故事板节点展示。
- 视频链路：`videoGenNode` 提交批量视频任务，`videoResultNode` 展示和选择结果。
- 场景构图：`sceneComposerNode` 作为特殊嵌入式节点，产出构图图像与提示词。
- 自动持久化：项目级快照和视口快照分开保存，桌面端走 SQLite，本地图片与视频引用做池化。

## 当前系统与旧版认知的最大区别

如果你以前看过这个项目，需要先更新三个认知：

1. 产品不再只是“分镜助手”，而是更完整的 Infinite Canvas 创作平台。
2. 节点体系已经从“生成节点直接出结果”演进成“生成节点 + 结果节点 + 派生链路”。
3. 持久化和桥接能力已经明显系统化，Web fallback 也不是完全空壳，但与桌面端仍不等价。

## 你应该记住的 6 个事实

1. **节点图是主交互，不是附属表达。** 很多需求本质是在定义上下游关系，而不是改按钮。
2. **`nodeRegistry.ts` 是节点系统的单一真相源。** 节点菜单、连线能力、默认数据都从这里出。
3. **结果节点是正式一等公民。** 图片和视频结果都被建模为独立节点，而不是生成节点里的临时字段。
4. **系统同时存在 live resolve 和 connection-time copy。** 这决定了“上游变化后下游是否自动感知”。
5. **桌面端是主形态。** 图片生成、部分本地文件处理、SQLite、系统能力依赖 Tauri/Rust。
6. **自动保存是系统能力，不是单个按钮。** `projectStore` 会根据整项目变更和视口变更走不同写入通道。

## 典型工作流

### 图片与视频

`uploadNode -> imageNode -> imageResultNode -> videoGenNode -> videoResultNode`

这条链路表达的是：上传参考图，做图像生成或编辑，产出一批图片结果，再从某个结果派生到视频生成，最后在视频结果节点里查看多候选视频。

### 剧本与分镜

`scriptUploadNode -> storyboardLlmNode -> storyboardGenNode -> storyboardNode`

这条链路表达的是：上传剧本，生成镜头提示词，批量产出分镜图，最后生成故事板结果节点。

### 构图辅助

`uploadNode / imageResultNode -> sceneComposerNode -> imageNode`

这条链路表达的是：用户把现有图片送入 Scene Composer 做构图，再把构图 prompt 或导出的图片喂给下游图像节点。

> **📌 PM 视角：** 这个项目最重要的管理抓手不是“页面”，而是“流”和“派生”。只要一个需求会改变节点如何取上游输入、如何产出结果节点、是否需要持久化恢复，它就已经不是单纯 UI 调整。
