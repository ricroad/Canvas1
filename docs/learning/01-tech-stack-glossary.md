# 技术栈术语表

## React

一句话定义：React 是用组件组织界面的前端库。

在本项目里的角色：负责应用外壳、画布页面、节点组件、对话框和设置界面。核心入口在 `src/App.tsx`，画布主编排在 `src/features/canvas/Canvas.tsx`。

> **📌 PM 视角：** React 让界面微调很快，但“能画出来”不等于“系统语义就改完了”。一旦需求碰到节点数据、桥接、持久化，成本就不再只是前端组件级。

## TypeScript

一句话定义：带类型系统的 JavaScript。

在本项目里的角色：约束节点数据结构、store 状态、command 参数和模型定义。节点类型集中在 `src/features/canvas/domain/canvasNodes.ts`。

> **📌 PM 视角：** TypeScript 会把改动影响面暴露得更早。例如你要加一个节点字段，经常会连带提示你去补 resolver、store、持久化和 i18n。

## Vite

一句话定义：前端开发服务器和构建工具。

在本项目里的角色：负责 `npm run dev`、前端打包，以及 `import.meta.glob` 自动发现 provider / model 文件。

> **📌 PM 视角：** 这也是为什么模型和 provider 扩展可以走“加文件 + 注册约定”的路线，而不是每次都改中心化大表。

## Zustand

一句话定义：轻量级状态管理库。

在本项目里的角色：`canvasStore` 管画布即时状态，`projectStore` 管项目生命周期与持久化协调，`settingsStore` 管用户设置，另外还有 `copilotStore`、`themeStore`。

> **📌 PM 视角：** 工程师说“这个要进 store”，通常代表它要跨组件共享、进入撤销重做、被持久化或参与更多派生逻辑。

## @xyflow/react

一句话定义：节点图 / 无限画布引擎，也就是 React Flow 的新包名。

在本项目里的角色：承载拖拽、连线、缩放、节点选择、边渲染、画布交互。

> **📌 PM 视角：** 它非常适合工作流型产品，但也意味着需求讨论必须考虑节点语义、连线策略、结果节点布局和批量操作，而不是只看“页面区块”。

## Tailwind CSS

一句话定义：原子化 CSS 方案。

在本项目里的角色：大多数前端视觉样式直接写在组件附近，全局尺寸和变量再由 CSS token 补充。

> **📌 PM 视角：** 调局部样式通常便宜，改全局尺寸体系、主题色或统一控件规范通常会更贵。

## Tauri 2

一句话定义：用系统 WebView 承载前端，用 Rust 提供本地能力的桌面容器。

在本项目里的角色：桌面端主运行时，前端通过 `src/commands/*.ts` 进入 Rust command。

> **📌 PM 视角：** 一旦需求涉及本地文件、SQLite、系统目录、剪贴板或更新检查，它基本已经越过纯前端边界。

## Rust

一句话定义：强调性能和安全的原生语言。

在本项目里的角色：实现 Tauri command、SQLite 读写、图片处理、模型请求、更新检查等桌面能力。

> **📌 PM 视角：** Rust 不是“附带后端”，而是桌面端真实能力的主要承载层。

## SQLite

一句话定义：嵌入式本地数据库。

在本项目里的角色：桌面端项目持久化后端，保存节点、边、历史、视口和脚本数据。

> **📌 PM 视角：** 这说明项目不是单页面临时态工具，而是有“项目资产”和“恢复能力”的桌面应用。

## IndexedDB

一句话定义：浏览器本地数据库。

在本项目里的角色：Web fallback 下替代 SQLite 保存项目记录。

> **📌 PM 视角：** Web 版不是完全没有存储能力，但它是降级实现，不等于桌面端拥有同级别文件和系统能力。

## i18next

一句话定义：多语言国际化框架。

在本项目里的角色：`src/i18n/index.ts` 负责加载 `zh.json` 和 `en.json`，节点和全局 UI 都通过 key 取文案。

> **📌 PM 视角：** 新功能不是只补中文文案，正式功能要同步中英 key，否则很容易在切语言时暴露半成品。

## `import.meta.glob`

一句话定义：Vite 提供的批量模块发现能力。

在本项目里的角色：自动扫描 `models/providers/*.ts`、`models/image/**/*.ts`、`models/video/**/*.ts`。

> **📌 PM 视角：** 它让“新增模型”更接近配置化扩展，但前提是约定命名和后端适配同步成立。
