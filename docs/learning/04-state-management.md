# 状态管理

## 当前状态体系

这个项目不是一个大一统 store，而是按职责拆开的多 store 结构。最重要的是三块：

- `canvasStore`：画布即时状态
- `projectStore`：项目生命周期和持久化协调
- `settingsStore`：用户设置与全局偏好

另外还有 `themeStore`、`copilotStore` 等辅助 store，但对核心工作流的影响没有前三者大。
`assetLibraryStore` 也属于辅助 store：它只保存资产库面板开关、分类和选择集，资产条目本身从画布节点实时扫描，不单独持久化一份媒体索引。

## `canvasStore` 管什么

它是节点图运行时的核心。

当前主要负责：

- `nodes` / `edges`
- 历史快照与撤销重做
- 节点增删改
- 连线与断线
- 结果节点创建与归并
- `imageResultNode` / `videoResultNode` 的选择和派生关系
- `SceneComposer` 的连线时图片注入
- 连接节点本体时的 handle 语义和视频首帧 / 尾帧输入关系

从源码看，`canvasStore.ts` 已经不只是“存状态”，还包含不少图操作规则，因此它是理解节点行为的关键文件之一。

## `projectStore` 管什么

它更像“项目外壳”和“自动保存调度器”。

当前主要负责：

- 项目列表
- 新建、打开、重命名、删除项目
- 当前项目信息
- 项目脚本内容 `scriptMd`
- 脚本来源文件名与导入时间
- `scriptAnalysisJson`
- 整项目持久化
- 视口独立持久化

关键点是：它不直接参与节点图业务规则，但负责把运行时状态稳定保存下来。

## `settingsStore` 管什么

它是“全局用户偏好”和“运行配置”的集合。

当前实际内容比很多人想象得多，包括：

- provider API key
- Kling access key / secret key / enabled
- 视频并发上限
- 分镜规划 skill
- 下载预设路径
- 是否显示节点价格
- 汇率与价格显示模式
- UI 圆角与主题色
- 画布边路由模式
- 自动检查更新开关

这说明 settings 不只是视觉偏好，而是会影响生成、成本展示和运行策略的全局输入。

## 为什么不把所有状态都塞进一个 store

因为三类状态粒度不同：

- 画布编辑态：高频变化
- 项目生命周期态：中频变化，但要持久化
- 用户设置：低频变化，但全局生效

如果全部混在一起，会带来三个问题：

1. 依赖扩散
2. 持久化责任混乱
3. 测试和定位问题困难

## 哪些状态不该进 store

通常不该进 store 的，是只服务当前组件的一次性局部状态，例如：

- 输入框临时草稿
- 弹层开关
- hover 态
- 纯视觉动画态

一旦这些状态被提升到 store，就会扩大订阅范围和耦合面。

## 哪些状态必须进 store

通常符合下面任一条件，就值得进 store：

- 多个组件要共享
- 会影响上下游节点行为
- 要进入撤销重做
- 要被项目持久化
- 要跨页面 / 跨会话恢复

## 当前系统里最容易低估的状态点

1. 结果节点选中项不是纯 UI，它会影响后续派生。
2. `generatedResultNodeIds` 不是装饰字段，它建立了生成节点和结果节点之间的关系。
3. `currentBatch` 不是瞬时 loading，它会影响轮询、恢复和失败状态。
4. `scriptMd` 与 `scriptAnalysisJson` 不属于节点内部，它们是项目级资产。
5. 资产库的 `selectedIds` 是 UI 操作态，不等于项目数据；真正的资产来源仍是节点图。

> **📌 PM 视角：** 当工程师说“这个状态要进 store”，你应该马上追问三件事：它归哪个 store、是否要持久化、是否要进入历史。只要其中两项答案是“要”，这个需求就不是轻量改动了。
