# AIGC 团队高效制作台 v2.0 PRD

> 版本：v2.0 草案 · 2026-04-24
> 背景：面向内部 AIGC 创作团队与后续接单生产需求
> 承接：当前 Canvas Creation Platform v0.1.13 桌面端基础版本
> 定位：先提高团队成员个人单体生产力，再逐步沉淀团队级资产与自动化能力

---

## 0. 一句话目标

把当前画布式生成工具升级为 **AIGC 创作团队的高效制作台**，让单个成员能用更少操作完成批量试图、结果筛选、资产复用和项目交付。

v2.0 的核心不是先做完整 Agent 平台，而是先把团队高频制作流程做丝滑：

- 少重复配置
- 少重复连线
- 少重复切模型
- 更快批量生成
- 更快比较结果
- 更好沉淀角色 / 场景 / 风格资产
- 更方便导出交付

---

## 1. 项目背景

### 1.1 团队现状

我们是一个 AIGC 创作团队，当前主要诉求是提高团队成员的个人产能。后续可能承接外部项目，因此工具不仅要能“生成”，还要能支撑更稳定的项目制作流程。

当前团队成员的真实工作不是单次 prompt 生成，而是：

1. 根据 brief / 剧本 / 参考图拆出多个 shot。
2. 为每个 shot 准备参考图、角色、场景和风格约束。
3. 反复切模型试图。
4. 从多个候选中挑选最佳结果。
5. 基于最佳结果继续做视频、构图修正或二次生成。
6. 整理资产并导出给客户或内部下游环节。

这些动作目前在画布上已经能完成，但重复操作偏多，流程不够连续。

### 1.2 当前产品能力

| 模块 | 当前状态 |
|---|---|
| 无限画布 | 已有节点、连线、缩放、拖拽、历史回退 |
| 图片生成 / 编辑 | 已有 `imageNode` + `imageResultNode` |
| 视频生成 | 已有 `videoGenNode` + `videoResultNode` |
| 分镜生成 / 分镜结果 | 已有 `storyboardGenNode` + `storyboardNode` |
| 构图修正 | 已有 `sceneComposerNode` |
| 资产库 | 已有左侧入口、分类、定位、下载 |
| 模型参数控件 | 已统一为 `ModelParamsControls` |
| 磁吸连接 | 已支持更大命中区和节点本体落点 handle 解析 |
| 桌面端能力 | Tauri + Rust + SQLite 已具备 |

### 1.3 当前主要痛点

做 5-20 个 shot 时，用户会遇到：

- 每个节点重复选择模型、比例、质量。
- 同类 shot 重复复制 prompt、参考线和参数。
- 想比较模型效果时，需要手动多次切模型、多次生成、多次找结果。
- 批量生成缺少统一调度和进度反馈。
- 生成失败后需要用户自己判断是否重试。
- 结果分散在画布上，缺少按模型、shot、批次、采用状态的管理。
- 角色 / 场景 / 风格参考无法很好复用到下一个项目。

## 2. 产品原则

### 2.1 用户只做决策动作

用户应该主要做：

- 选择方向
- 调整 prompt
- 选择模型组合
- 审核结果
- 采纳 winner
- 导出交付

系统应该尽量承担：

- 继承参数
- 自动接线
- 批量调度
- 自动重试
- 结果归组
- 资产整理

### 2.2 先做工作台，再做 Agent

v2.0 不以完整 Agent 自动创作为第一目标。原因：

- 团队当前最需要的是高频制作流程提效。
- Agent 需要依赖稳定的资产、模板、结果和批量调度基础设施。
- 内部团队先跑出稳定 SOP，再让 Agent 学习和自动化这些 SOP。

### 2.3 画布仍然是核心工作区

所有自动化能力都不应该绕开画布，而应该增强画布：

- 自动创建节点
- 自动连接节点
- 自动填参数
- 自动生成结果节点
- 自动沉淀资产

## 3. 目标用户

### 3.1 核心用户

| 用户 | 需求 |
|---|---|
| AIGC 视觉创作者 | 快速试图、选图、二次生成 |
| 分镜 / 短剧制作成员 | 批量创建 shot，保持角色和场景一致 |
| 视频生成操作者 | 从已选图片快速生成视频候选 |
| 项目负责人 | 管理项目资产、结果和交付物 |

### 3.2 非目标用户

v2.0 暂不优先服务：

- 外部 SaaS 多租户用户
- 协同编辑用户
- 无画布概念的新手消费用户
- 完全自动生成整片且无需人工审核的 Agent 用户

## 4. 成功指标

### 4.1 效率指标

| 指标 | 当前预估 | v2.0 目标 |
|---|---:|---:|
| 5-shot 首轮出图操作次数 | 50+ | <= 25 |
| 新建同类 shot 平均操作次数 | 8-12 | <= 3 |
| 同一 shot 多模型试图次数 | 手动 N 次 | 1 次 |
| 从结果图生成视频操作次数 | 5-8 | <= 2 |
| 批量生成失败后的恢复操作 | 手动判断 | 一键重试 / 自动重试 |

### 4.2 质量指标

- 图片擂台中，用户至少 60% 的生成会采纳一个 winner。
- 用户能清楚看到每个结果来自哪个模型。
- 批量生成过程中，失败项不会阻塞其它项。
- 结果节点能承载完整的模型、参数、批次、成本信息。

### 4.3 成本指标

- 擂台生成前必须展示预计调用模型数、预计结果数和成本提示。
- 视频擂台默认关闭，开启必须二次确认。
- 批量生成必须尊重 provider 并发限制。

## 5. v2.0 MVP 范围

v2.0 MVP 聚焦一个闭环：

`项目默认参数 -> 复制 shot -> 图片擂台 -> 结果分组对比 -> 采纳 winner -> 一键派生视频 / 下载交付`

### 5.1 P0 功能

| 编号 | 功能 | 目标 |
|---|---|---|
| P0-1 | 项目级默认参数 | 减少每个节点重复配置 |
| P0-2 | 复制为新 shot | 快速复用上一 shot 的结构、参考线和参数 |
| P0-3 | 图片擂台 | 一次调用多个图片模型生成候选 |
| P0-4 | Result 节点分组对比 | 按模型展示候选并选择 winner |
| P0-5 | 批量生成调度器 | 一键生成多个准备好的节点 |
| P0-6 | 自动重试 | 降低瞬时 API 失败对用户的打断 |

### 5.2 P1 功能

| 编号 | 功能 | 目标 |
|---|---|---|
| P1-1 | Prompt AI 帮写 | 基于参考图和上下文生成 prompt 草稿 |
| P1-2 | 资产库升级 | 支持收藏、采用状态、按批次 / 模型筛选 |
| P1-3 | 项目模板 | 快速创建常见工作流 |
| P1-4 | 视频快捷派生 | 从 winner 图片一键创建视频生成节点 |

### 5.3 暂不进入 v2.0 MVP

- 完整 Series / Episode 系统
- Bible 自动生成与维护
- 多 Critic 审查系统
- Visual Critic / Continuity Critic
- Web 多用户系统
- 用户自定义 Skill 市场
- 视频擂台默认开放

## 6. 功能详述

## 6.1 项目级默认参数

### 目标

用户创建新节点时自动继承项目默认模型、比例、质量和输出数量，减少重复配置。

### 行为

项目设置中新增“生成默认值”：

- 图片默认模型
- 图片默认比例
- 图片默认 quality / size
- 图片默认输出数量
- 视频默认模型
- 视频默认时长
- 视频默认比例
- 视频默认输出数量

节点创建时读取项目默认值。节点仍可单独覆盖。

### 验收

- 新建 `imageNode` 自动带入项目图片默认参数。
- 新建 `videoGenNode` 自动带入项目视频默认参数。
- 修改节点参数不会反向污染项目默认值。
- 项目默认值随项目保存和恢复。

---

## 6.2 复制为新 shot

### 目标

让用户快速基于上一 shot 创建下一 shot，只改差异，不重复搭节点。

### 行为

节点右键新增：

- 复制为新 shot
- 复制选中节点为新 shot 组

复制内容：

- prompt
- 模型
- 比例
- quality
- extra params
- 上游参考线
- 结果派生关系不复制
- 运行态不复制

### 验收

- 复制后的节点保持参考图连接。
- 复制后的节点不继承 `isGenerating/currentBatch/generatedResultNodeIds`。
- 复制后节点位置自动错开，不遮挡原节点。

---

## 6.3 图片擂台

### 目标

把“反复切模型生成比较”的流程压缩为一次操作。

### UI

在 `imageNode` 参数区增加“擂台模式”开关。

开启后：

- model chip 从单选变多选。
- 最多选择 4 个模型。
- 每个模型默认生成 1 张。
- 高级设置允许调整“每模型生成数量”。

### 生成规则

一次擂台生成 = 多个模型任务。

共享参数：

- prompt
- 参考图
- 比例
- 通用 quality

模型专属参数：

- 默认使用各模型自己的默认值。
- P1 再支持 per-model 编辑。

### 结果规则

所有结果进入同一个 `imageResultNode`。

每个 variant 需要记录：

```ts
{
  variantId: string;
  imageUrl: string;
  modelId: string;
  providerId: string;
  contestId?: string;
  contestGroupId?: string;
  createdAt: number;
  snapshotParams: Record<string, unknown>;
  retryCount?: number;
  errorMessage?: string;
}
```

### 失败隔离

- 单模型失败不影响其它模型。
- 失败模型在结果节点中显示失败卡片。
- 用户可单独重试失败模型。

### 成本提示

点击生成前展示：

- 选择了几个模型
- 每个模型生成几张
- 总结果数
- 预计成本区间
- 哪些模型缺 API key

### 验收

- 开启擂台后可选择 2-4 个图片模型。
- 生成后同一个 `imageResultNode` 按模型分组展示结果。
- 用户能设置 / 切换 winner。
- 下游派生读取当前 winner。
- 单模型失败不会中断其它模型结果入库。

---

## 6.4 Result 节点分组对比

### 目标

让结果节点成为“候选池”，而不是单纯图片容器。

### 行为

`imageResultNode` 增加两种视图：

- 全部平铺
- 按模型分组

每个候选展示：

- 缩略图
- 模型名
- provider
- 生成时间
- 成本估算
- 采纳状态

操作：

- 设为 winner
- 继续生成图片
- 生成视频
- 构图修正
- 下载
- 加入资产库收藏

### 验收

- 用户能清楚知道每个结果来自哪个模型。
- winner 状态持久化。
- 从结果派生时使用 winner 对应图片。

---

## 6.5 批量生成调度器

### 目标

让用户一次性生成多个准备好的节点，而不是逐个点击。

### 入口

画布顶部新增：

- 全部生成
- 生成选中节点
- 只生成 AI 草稿

### 调度规则

- 只调度未生成 / 可重新生成的节点。
- 尊重 provider 并发限制。
- 图片、视频分队列。
- 视频队列默认更保守。
- 失败项不阻塞其它项。

### 状态展示

顶部进度条显示：

- 总任务数
- 已完成
- 进行中
- 失败
- 当前 provider 队列
- 预计剩余

### 验收

- 批量生成时用户能看到全局进度。
- 单节点失败不影响其它节点。
- 达到并发上限时进入等待，而不是直接失败。

---

## 6.6 自动重试

### 目标

把瞬时错误从用户操作中拿掉。

### 适用错误

- 5xx
- 超时
- 限流
- provider 临时不可用

### 不自动重试

- 缺 API key
- 余额不足
- 参数非法
- 图片格式不支持
- 内容安全拒绝

### 规则

- 最多自动重试 3 次。
- 指数退避。
- 结果节点显示“已自动重试 N 次”。
- 用户可手动重试失败项。

## 7. P1：资产与模板

## 7.1 资产库升级

当前资产库从节点扫描资产。P1 升级为项目制作管理入口。

新增能力：

- 收藏
- 已采用 / 未采用
- 按模型筛选
- 按生成批次筛选
- 按 shot 分组
- 拖回画布创建节点
- 一键下载当前采用结果

## 7.2 角色 / 场景资产

先不做复杂 Series Bible，只做轻量资产实体：

- Character Asset
- Scene Asset
- Style Asset

它们可以由上传图、结果图或构图结果转化而来。

## 7.3 项目模板

MVP 模板：

- 单图生成
- 图片擂台
- 5 格分镜
- 角色三视图
- 图生视频
- 构图修正到图片生成

---

## 8. P2：Agent 方向

Agent 是后续方向，但应建立在 v2.0 的工作台能力之上。

未来 Agent 可以做：

- 自动拆 brief / 剧本
- 自动生成 shot 列表
- 自动创建画布节点
- 自动接角色 / 场景资产
- 自动写 prompt
- 自动推荐模型组合
- 自动质检一致性

但 v2.0 不以完整 Agent 流程为主线。

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 擂台成本不可控 | 用户一次点击消耗过高 | 生成前成本确认、模型数上限、默认每模型 1 张 |
| 多模型参数不兼容 | 生成失败率上升 | MVP 只共享通用参数，模型专属参数用默认 |
| 批量生成导致限流 | 队列卡顿 | per-provider 队列和并发限制 |
| 结果节点过载 | 候选太多难比较 | 默认按模型折叠分组，只突出 winner |
| 视频生成慢且贵 | 体验拖累 | 视频擂台不进 MVP，视频队列单独限速 |
| 资产体系过早复杂化 | 开发周期拉长 | P1 先做轻量资产，不做完整 Series/Bible |

## 10. 里程碑

### M1：工作流提效基础（1-2 周）

- 项目级默认参数
- 复制为新 shot
- 自动重试

### M2：图片擂台闭环（2-3 周）

- 图片擂台开关
- 多模型提交
- Result 节点分组
- winner 机制
- 成本提示

### M3：批量生成调度（2-3 周）

- 全部生成
- 选中生成
- 队列与并发
- 全局进度
- 失败项重试

### M4：资产与模板（3-4 周）

- 资产库增强
- 收藏 / 采用状态
- 轻量角色 / 场景 / 风格资产
- 项目模板库

### M5：Agent 预研（独立评估）

- 基于真实生产数据定义 Agent 自动化边界
- 再写 Agent 专项 PRD

## 11. v2.0 非目标

- 不做复杂 Bible / Series 系统。
