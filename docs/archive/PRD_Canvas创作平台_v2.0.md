# Canvas Creation Platform v2.0 PRD

> 版本：v2.0 草案 · 2026-04-24
> 承接：当前 v0.1.13（已交付的桌面端基础版本）
> 输出物：本文为产品方向文档，技术拆解 / 排期 / 接口契约由技术团队基于本文继续细化。

---

## 0. 一句话目标

把当前"画布式生成工具"升级为"剧本驱动的 agent 编排生成平台"：

- 用户上传剧本 → 系统自动拆集 / 分析 / 预置画布 / 写 prompt / 调度多模型生成
- 用户只做"选择 / 微调 / 审核 / 采纳"这些**决策性动作**，不再做重复的**操作性动作**
- 同时保留当前画布作为"手动精修"的入口，agent 自动化与手动编辑并存

## 2. 总体路线图

两条正交战线并行：

```
战线 A：现状工作流优化（面向已有用户，立刻见效）
       α → β → γ  三阶段

战线 B：Agent 系统（面向核心用户价值重构）
```

## 3. 战线 A：现状工作流优化

### 3.1 α 阶段 · 低垂果实（约 1 周）

目标：把 50 次点击降到 ~35 次。

| 编号 | 优化项 | 行为变化 |
|---|---|---|
| **α-1** | 项目级默认参数 | 项目设置里可设"默认模型 / 默认比例 / 默认 quality"；新建节点自动继承；节点上仍可覆盖 |
| **α-2** | 节点"复制为新 shot" | 右键菜单"复制为新 shot"——复制 prompt、模型、参数、已连参考线；支持多选节点批量改某字段 |
| **α-3** | 自动重试 | 生成 API 瞬时失败（5xx、限流、超时）自动重试 3 次指数退避；Result 节点可见"已自动重试 N 次" |

### 3.2 β 阶段 · 中层改造（约 2-3 周）

目标：引入 agent 能力雏形，**β-2 / β-3 直接复用到战线 B**。

| 编号 | 优化项 | 行为变化 | 与战线 B 关系 |
|---|---|---|---|
| **β-1** | 参考线继承 | 新建 ImageEdit / VideoGen 节点时，右键"继承上一个同类节点的参考线"；或拖出节点时，画布上若有最近使用的 upload 节点自动提示绑定 | 未来 F6 自动接线的前身 |
| **β-2** | Prompt "✨ AI 帮写" 按钮 | 节点 prompt 框右上角按钮，读取连线的参考图 + 节点上下文调 LLM 生成 prompt 草稿；用户可编辑后生成 | 未来 F8 / 第一个内置 Skill |
| **β-3** | "全部生成 (N)" 批量按钮 | 画布顶部按钮，调度所有"未生成 / AI 草稿已就绪"的节点生成；尊重 per-provider 并发限制；顶部挂进度条 | 未来 agent 一键流的基础 |

### 3.3 γ 阶段 · 数据模型（约 2-3 周）

目标：为战线 B 铺基础设施，同时解决当前用户的资产复用痛点。

| 编号 | 优化项 | 行为变化 |
|---|---|---|
| **γ-1** | 角色 / 场景资产实体化 | 把 UploadNode 升格为"资产引用"；Character / Scene 是 Series 级的独立资产表，跨集 / 跨项目可复用；画布节点只是资产的 reference |
| **γ-2** | 项目模板库 | 新项目时可选模板（单图生成 / 5 格分镜 / 角色三视图 / 视频转场 / 剧本驱动）；模板 = 预置节点 + 预置资产槽位 |

⚠️ **γ-1 是战线 B 的前置条件**，强烈建议独立一期先交付。

### 3.4 跨阶段专项：擂台模式（约 2 周）

一次调用多模型生成，结果按模型分组叠放。**与 β-3 共享"批量调度器"基础设施**，两者绑定做。

#### 3.4.1 核心设计（✅ 已拍板）

- **UI 模式**：在现有 ImageEdit / VideoGen 节点上加"擂台模式开关"。model chip 变成多选（上限 4 个）。产出塞到同一个 Result 节点，按模型分组展示
- **失败隔离**：单个模型失败不影响其它；失败项可独立重试
- **视频擂台**：默认关闭，开启时弹硬性成本确认

#### 3.4.2 待确认决策 ❓

| 决策项 | 推荐 |
|---|---|
| 参数跨模型处理 | 只共享通用项（prompt / refs / 比例），其他用各自默认；给高级用户一个"per-model 编辑"折叠入口 |
| Result 节点布局 | 按模型分组，每组横向排 variants |
| 胜者机制 | 保留手动采纳；增加"冠军"标签用于统计 |
| 成本可见性 | 首次点生成前弹预估；之后画布顶部常驻"本轮累计"账单条 |
| 预设模型组合 | MVP 每次手选；P1 支持保存"我的擂台组合" |

#### 3.4.3 数据沉淀

每次擂台产出一条记录：`{ shot 类型, 模型列表, 采纳模型, 时间戳 }`。

- MVP：数据只展示给用户本人（"你最常采纳 Nano Banana Pro"）
- P1：基于历史的"推荐模型"按钮
- P2：未来 agent 系统的 Visual Critic 数据源

## 4. 战线 B：Agent 系统

### 4.1 整体数据模型

两级实体，上层 = 全剧共享资料，下层 = 单集工作空间：

```
Series (剧)
├─ bible.md              ← LLM 通读后产出的核心资料，用户可编辑
├─ characters[]          ← 由 γ-1 支持
├─ scenes[]              ← 由 γ-1 支持
└─ episodes[]
     ├─ script           ← 原文 + 行号范围
     ├─ analysis         ← 单集 LLM 分析结果（shots[], 每 shot 的人物/场景引用）
     ├─ canvas           ← 本集画布（nodes, edges, viewport）
     └─ results          ← 本集生成的图/视频产物
```

**Bible 和 Episodes 共用同一份 Character/Scene 资产表，通过名字 JOIN**。

### 4.2 流水线（F1-F9）

```
F1 剧本上传
  ↓
F2 正则拆集（✅ 不用 LLM）
  ↓
F3 LLM 通读 → bible.md 草稿
  ↓ 侧栏可查/改；异常时顶部提示条
F4 进入 Series 主页，看到集列表
  ↓ 点某一集
F5 LLM 集级分析（输入 = bible + 单集剧本）
  ↓ 产出 shots[]，每个 shot 标注 characters[] + scene
F6 自动预置画布：人物/场景资产节点 + 分镜生成节点
  ↓
F7 自动接线（纯名字 JOIN，✅ 规则引擎，不用 LLM）
  ↓
F8 LLM 为每个 shot 写分镜 prompt 并填入节点（Skill 调用）
  ↓ 节点显示黄色"AI 草稿"边框
F9 用户审完 → 点"全部生成"→ 批量调度生成
```

### 4.3 Bible 机制

#### 4.3.1 Bible.md 结构（强 schema 的 Markdown）

```markdown
## 核心设定
## 主题与基调
## 人物
  ### 李明
  - 别名: 小李, 李主任
  - 形象: ...
  - 弧光: ...
## 场景
  ### 客厅
  - 外观: ...
## 主线走向
- 第 1-3 集: ...
```

二级标题是机器锚点；三级标题是人物/场景单元。

#### 4.3.2 Bible 生命周期（✅ 不强制审核）

- 生成后自动进入"已就绪"状态
- 侧边栏永驻 bible 编辑入口
- 仅在**高置信度异常**时顶部提示条提示（见 §4.6）
- 首次生成时弹一次"AI 已起草 bible"引导，可勾"不再提醒"
- Bible 修改后已分析的集**不自动重算**，只在该集顶部显示"bible 已变更，建议重新分析"

#### 4.3.3 单集 LLM 调用形态

```
system: 这是 {series_title} 的 bible。后续分析必须引用 bible 中的人物主名和场景主名。
        <bible.md 全文>

user:   本集剧本：
        <episode_N.txt>

        请输出 JSON schema：
        { shots: [ { id, content, characters: string[], scene: string, modifiers: string[] } ] }
```

- `characters[]` 必须引用 bible 中的 **主名**（代词 / 别名先由 LLM 归一）
- `scene` 引用 bible 中的场景主名；光线 / 时段 / 天气等变化进 `modifiers[]`（✅ 场景 3B 方案）

### 4.4 自动接线规则（F7）

```python
for shot in episode.shots:
  characters_wired = []
  for name in shot.characters:  # 主名
    asset = series.characters.find(name)  # 严格匹配
    if asset:
      wire(asset.node, storyboardGen.refs[i])
      characters_wired.append(asset)
    else:
      # ✅ 缺资产兜底：占位节点
      wire(placeholder("待补充: " + name), storyboardGen.refs[i])

  scene = series.scenes.find(shot.scene)  # 严格匹配
  if scene:
    wire(scene.node, storyboardGen.scene_input)
```

- ✅ 严格匹配，不走模糊
- ✅ 缺资产 → 自动占位节点，标红提示
- **一个 shot 参考图超过 storyboardGen 支持数**：按"台词量/戏份权重"排序取前 N，其余仅进 prompt 文本

### 4.5 Skill 系统

#### 4.5.1 形态（❓ 推荐 3A）

**MVP 硬编码若干内置 skill，不开放用户自定义**。Skill 就是"带约定输入输出的 LLM 调用"，在前端/后端注册，节点可调用。

#### 4.5.2 首批内置 Skill

| Skill | 职责 | 对应 feature |
|---|---|---|
| `writeShotPrompt` | 输入 shot 上下文 + 参考资产 → 输出分镜 prompt | F8 / β-2 |
| `enhancePrompt` | 输入用户草稿 → 输出改写版 | β-2 的改写按钮 |
| `bibleCritic` | 输入 bible 草稿 → 输出 issues[] | §4.6 Critic 层 |
| `shotConsistencyCritic` | 输入单集分析 → 输出 issues[] | §4.6 Critic 层 |
| `promptCritic` | 输入 prompt → 输出 issues[] | §4.6 Critic 层 |
| `visualCritic` | 输入生成结果 + 参考 → 输出一致性评分 | §4.6 Critic 层 |
| `continuityCritic` | 输入整剧生成结果 → 输出跨集矛盾 | §4.6 Critic 层（P2） |

### 4.6 Review 体系

核心原则：**Agent 永远不卡住用户流程**。所有 review 都是"可感知、可 skip"。

#### 4.6.1 5 种 Review 通道（按场景叠用，非互斥）

| 通道 | 形态 | 打扰度 | 场景 |
|---|---|---|---|
| ① 阻塞式 modal | 弹窗必操作 | 极高 | 灾难级：API 失败、额度耗尽 |
| ② 顶部提示条 | 不遮挡的一条 | 中 | 系统性问题（bible 缺字段等） |
| ③ 首次引导 | 首次触发弹一次，可永久关 | 中（仅首次） | 关键操作初见面 |
| ④ 就地徽标 / 边框色 | 节点视觉标记 | 低 | 单节点级（AI 草稿 / 缺资产 / 敏感词） |
| ⑤ 被动可查侧栏 | 永远挂在侧边 | 零 | bible.md / 人物表 / 场景表 |

#### 4.6.2 节点边框视觉语言

| 颜色 | 含义 |
|---|---|
| 灰色 | 正常 |
| 黄色虚线 | AI 生成未审核 |
| 红色 | 有错误（缺资产 / 敏感词 / 生成失败） |
| 绿色脉动 | 正在跑 |

#### 4.6.3 Subagent Critic 层（上游发动机）

Critic 不直接跟用户说话，它输出给规则层决定走哪条通道：

```
LLM-A 生成 → Critic 审查 → { verdict, issues[] }
                              ↓
                ┌────────────┼────────────┐
              pass         flag          retry
             (静默)    (喂入原 5 通道)  (自动重跑 A 带反馈)
```

5 个 critic（见 §4.5.2）的 verdict 规则、retry 上限（1 次）、可靠度等级在技术文档单独展开。

#### 4.6.4 Review 密度三档

用户设置里可调（默认"引导模式"）：

| 档位 | 行为 |
|---|---|
| 引导模式 | 所有 ③ 首次引导全开；② 提示条全开 |
| 标准模式 | 只保留 bible 的 ③；② 全开；④ 全开 |
| 专家模式 | ③ 全关；② 只在阻塞级；④ 徽标全开；① 永远不能关 |

**正交维度：Critic 开关**（全开 / 基础 / 关闭），允许用户关掉贵的 Visual/Continuity Critic 省成本。

#### 4.6.5 4 个 Checkpoint 的具体形态

| Checkpoint | 通道组合 | 强制度 |
|---|---|---|
| ① Bible | ⑤ 侧栏 + ③ 首次引导 + ② 异常时提示条 | 非强制 |
| ② 集分析 | ④ 画布顶部摘要条（可折叠）+ ③ 首次引导 + ② 异常时 | 非强制 |
| ③ 分镜 prompt | ④ 黄色边框 + 生成时 ①（可关）+ 批量审查抽屉 | 强建议 |
| ④ 生成结果 | ④ 徽标（采纳 / 丢弃 / 质检不过）+ 行内 toast | 常规 |

**时机未定 ❓**：推荐战线 A + B 在桌面跑通后再搬（7A）。

## 8. 风险与未解决项

### 8.1 技术风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| 长剧本 LLM 通读超 token 限制 | F3 bible 生成不稳 | map-reduce 分片 → 合并；或分章节递进 |
| 结构化输出失败率 | F3/F5 整套流水线脆弱 | 带 schema 校验 + 最多 2 次带反馈 retry；最后 flag 人工 |
| 多 provider 并发限流 | 擂台 / 批量生成卡顿 | per-provider 队列 + 可视化进度 |
| 人物视觉一致性（同一角色跨 shot 长相不同） | Agent 体验致命缺陷 | P2 引入 IP-Adapter / LoRA / 强参考约束；MVP 只做警告 |
| 成本失控（擂台 + Critic + 批量） | 用户消费超预期 | 顶部常驻账单条 + 预算上限设置 + critic 可关 |

### 8.2 产品决策悬而未决

- ❓ 6 个技术分叉点（§6）等拍板
- ❓ Copilot 侧栏未来定位（转型为 Agent 控制台 vs 退役）
- ❓ 视频擂台是否进 MVP
- ❓ Web 化时机
- ❓ 剧本格式规范细节（支持 `EP X` / `第X回` / 等变体？）

### 8.3 产品决策已定 ✅

- 集拆解用正则工具而非 LLM
- Bible 审核非强制
- 场景命名用单资产 + modifier
- 匹配策略严格 + 人工 review 兜底
- Review 多通道并存
- Subagent Critic 作为上游发动机
- 擂台 UI 用节点开关模式
- 画布节点的视觉语言（灰/黄/红/绿）

## 9. 附录：剧本格式规范 v1（✅ MVP 锁死版）

交给编剧团队的输入契约：

- 集标题必须独占一行，行首（前可有空格），格式：`第X集` 或 `第一集`（中文/阿拉伯数字都支持）
- 集前内容（前言 / 人物表 / 大纲）进 `preamble` 字段，不归属任何集
- 集内 `场` / `镜头` 不由正则处理，交给 LLM 分析
- 无集数标记 → 报错提示补充
- 不支持 `Episode X` / `第X回` / `EP X` 等变体（P2 再放宽）
